Chapter 02 · The Pipeline

传统 RAG 全链路：每一环的机制、数字与陷阱

本章是全系列的重心。把六步流水线拆成八个技术决策点，每个决策点给出机制解释、实证数字和生产默认值——以及一个反复出现的教训：简单基线出奇能打。

chunking recursive 512 检索 hybrid + RRF 重排 cross-encoder 评估 recall@k + RAGAS

§1全链路总览

把第 1 章的六步流水线展开成工程视角的完整管道。灰色是数据流，橙色节点是本章的八个决策点：

教训本章反复出现的模式

每一环都存在「简单基线出奇能打」的实证：固定大小 chunking 打平语义 chunking、BM25 反杀 dense 检索、RRF 不需要调权重。正确的姿势不是堆高级技巧，而是先搭最简管道 + 评估集，再评估驱动地逐环升级。

§2Chunking：切分策略

Chunk 是检索的原子单位，切分决定了「一次检索能拿到的语义完整度」。核心矛盾：块小 → 检索精准但语境碎；块大 → 语境全但 embedding 语义被稀释、更容易检入噪声。

策略谱系

策略机制适用成本
Fixed-size按 token 数硬切 + overlap（常 10–20%）兜底基线，效果意外地好零
Recursive按分隔符层级递归切（段落→句子→词），LangChain 默认生产默认，通用文本零
Document-based按 Markdown 标题 / 代码结构等文档固有边界切结构化文档（需好的解析，见第 3 章）零
Semantic按相邻句 embedding 相似度的断点切理论美好，实证不划算（见下）embedding 调用
Late chunking长上下文模型编码整篇后在 token embedding 层面切块池化跨块指代严重的文档长上下文 embedding
ContextualLLM 为每个 chunk 生成 50–100 token 语境前缀再嵌入高价值语料，愿付索引期 LLM 成本每 chunk 一次 LLM

实证数字：打破「越智能越好」

  • NAACL 2025 Findings：固定 200 词的块打平或超过语义分块，语义分块的额外计算不划算。
  • Vecta 2026.02 对 7 种策略 × 50 篇学术论文的基准：recursive 512-token 以 69% 准确率排第一，semantic chunking 只有 54%。
  • Anthropic Contextual Retrieval（2024.09）：语境前缀 + BM25 使 top-20 检索失败率从 5.7% 降到 2.9%（−49%），再叠加重排序共降 67%。用 prompt caching 把索引成本压到可接受。来源
  • Snowflake 金融 RAG 实证：解析输出层级结构后做 markdown-aware chunking，比固定/语义切分高 5–10 个百分点——切分质量的上限由上游解析决定（第 3 章展开）。
默认起手配置

recursive 切分、400–600 token、10–15% overlap、保留标题面包屑作为 chunk 元数据。有了评估集之后，再考虑 contextual retrieval（提升最实证）或 document-based（若解析质量高）。

§3Embedding：模型选型

Embedding 模型决定「语义相似」的定义。选型三问：多语言吗？自托管吗？维度预算多少？

模型MTEB 多语言特点
Qwen3-Embedding 8B/4B/0.6B70.58（8B，2025.6 登顶）开源，维度 32–4096 可调（MRL），中文强
Cohere embed-v4≈65.2商用 API，多模态输入
OpenAI text-embedding-3-large≈64.6生态默认，支持维度截断（3072→256）
BGE-M3≈63.0开源经典：单模型同时输出 dense / sparse / multi-vector 三种表示，做 hybrid 检索的天然选择
Voyage（voyage-4 系）—检索专精，被 MongoDB 收购，部分 Apache 2.0

两个工程要点

Matryoshka（MRL）：训练时让向量的前缀维度也可独立使用，推理时可直接截断（如 3072→256）换 12 倍存储和更快的 ANN，精度损失很小。OpenAI text-embedding-3 与 Qwen3-Embedding 都支持——向量库成本敏感时先试截断，再考虑换小模型。

MTEB 榜有过拟合嫌疑：榜单高分不保证你的语料好用。正确姿势：从自己的语料抽 100–200 个「查询 → 应命中文档」对，直接测 recall@k。这个小评估集也是后面所有环节的基础设施。

§4索引：HNSW / IVF 与向量库选型

百万级以上向量做精确暴力检索太慢，需要近似最近邻（ANN）索引。两大主流结构：

向量库选型

产品定位选它当
FAISS库，不是服务（无持久化/过滤/多租户）离线实验、算法研究
pgvectorPostgres 扩展，HNSW/IVFFlat 都支持已有 Postgres 时的默认选择——零新增组件，事务与业务数据同库
QdrantRust 实现，过滤检索强复杂 metadata 过滤 + 性能敏感
Milvus分布式，面向十亿级真正的大规模多租户场景
Chroma嵌入式，开发体验优先原型与 demo

本地案例：gbrain 用 PGLite/Postgres + pgvector（HNSW），LightRAG 默认 FAISS + 可插拔 15 种后端——第 5、8 章会看到它们的取舍。

§5检索：Dense / Sparse / Hybrid 与 RRF

Dense 和 sparse 的失败模式恰好互补，这是 hybrid 成为生产默认的根本原因：

Dense（向量）Sparse（BM25 / SPLADE）
擅长同义改写、跨语言、模糊语义（"怎么退货" ↔ "退款流程"）精确词面匹配：型号、错误码、函数名、专名（"ERR_4032"、"HNSW"）
失败低频专名被稀释；训练分布外的领域词同义词完全失配；语序无感
SPLADElearned sparse：transformer 在词表维度上学出稀疏权重——保持倒排索引的可解释与速度，同时获得语义扩展能力

RRF：不调参的融合

两路检索的打分量纲不可比（余弦相似度 vs BM25 分数），直接加权很脆弱。Reciprocal Rank Fusion 只用排名：

# k=60 是标准默认；偏重 top-1 精度可调至 30–40
score(d) = Σ每路检索 i  1 / (k + ranki(d))

文档在任一路排名靠前都能获得高融合分，且对单路的异常分数天然免疫。Elasticsearch、Weaviate、Qdrant、Vespa 均已内置。本地案例 gbrain 把 RRF 用到了四路：向量 + BM25 + 图关系遍历 + 源偏置（第 8 章）。

机制为什么 hybrid 几乎总是赢

检索失败大多是「某一类查询在某一路上系统性失败」：全 dense 时错误码查询全军覆没，全 BM25 时口语化查询全军覆没。Hybrid 的收益不是平均分提高几个点，而是砍掉最差情况——这在生产环境里比平均分重要得多。

§6重排序：Cross-Encoder 与 ColBERT

第一阶段检索（bi-encoder）为了速度把 query 和文档独立编码成单向量，天然损失交互信息。重排序用更贵的模型对 top 50–100 候选做第二阶段精排：

为什么 Bi-Encoder 有天花板：信息瓶颈

先看清一阶段检索输在哪。双塔把一段 500 token 的文档压进一个定长向量——「文档第 37 个词是否精确匹配了 query 里那个型号」这种 token 级信息在池化时就丢了。且 q 和 d 编码时互相看不见对方：文档向量是「对所有可能查询的平均妥协」，不是针对这个 query 的表示。三种架构的差异本质是一个问题：q 与 d 的交互发生得多早——越早越准，越晚越快。

Cross-Encoder：交互发生在每一层 attention

[CLS] query [SEP] document [SEP] → transformer → CLS 向量 → 线性层 → 标量分数

从第一层 attention 开始，query 的每个 token 就能和文档的每个 token 直接交互，所以它能捕捉双塔捕捉不到的三类信号：精确匹配感知（"ERR_4032" 直接对上）、否定与条件（「不支持 Windows」vs「支持 Windows」在单向量空间几乎重合）、词序与角色（「A 收购 B」vs「B 收购 A」）。

训练：MS MARCO 类标注数据上做二分类/回归 + 难负例挖掘——负例取 BM25/双塔检回的「像但不对」的文档，逼模型学细粒度区分；现代 reranker（BGE-reranker-v2-m3、Qwen3-Reranker、zerank）普遍再叠 LLM 蒸馏。

为什么只能做第二阶段：没有任何东西可预计算——100 万文档理论上要跑 100 万次完整前向。所以它的位置永远是：便宜的一阶段召回 top 50–100 → 逐对打分精排 → top 5–10 进 LLM。50–100 次前向共 <200ms，换 nDCG@10 提升 5–15 个点（词面难例 20+），是全管道性价比最高的一段。选型：托管 Cohere Rerank 3.5，自托管 BGE-reranker-v2-m3，两者 benchmark 接近。

ColBERT：Late Interaction，交互推迟但不取消

ColBERT（Khattab & Zaharia, SIGIR 2020）的洞察：双塔丢信息是因为池化成单向量，cross-encoder 慢是因为交互发生在编码期——那就把两者拆开：编码独立进行（文档侧可预计算），交互推迟到打分时，但保留 token 粒度。

离线：文档编码后不池化，每 token 保留一个向量（投影到 128 维省空间），300 token 的文档 = 300×128 矩阵。在线：query 逐 token 编码后用 MaxSim 打分：

# query 的每个 token 去文档里找一个最像自己的 token，把最佳匹配分求和
score(q, d) = Σi  maxj ( qi · dj )

直觉："ERR_4032" 这个 token 精确命中文档里的同名 token（拿到接近满分），"退款" 软匹配到 "退货款项"——精确匹配和语义匹配在同一个机制里自然共存，这是它在标识符密集查询上常反超 cross-encoder 的原因。

代价与解法：每文档从 1 个向量变成几百个，v1 索引大到不实用。ColBERTv2 用残差压缩解决——token 向量聚类后只存「最近质心 ID + 低比特量化残差」，索引缩小约一个数量级；配套 PLAID 引擎用质心剪枝，使它可直接当第一阶段检索器（不只是 reranker），延迟压到几十毫秒。

三者对照

Bi-EncoderColBERT（late interaction）Cross-Encoder
交互发生在从不（各编各的）打分时，token 级 MaxSim编码期，每层 attention
文档侧可预计算✅ 1 向量✅ n 向量（压缩后）❌ 完全不能
单 query 成本1 次编码 + ANN1 次编码 + MaxSim（可索引加速）k 次完整前向
精度基线接近 cross-encoder最高
存储1×压缩后约 5–10×0（无索引）
典型角色一阶段召回一阶段检索或轻量精排top-100 精排
记忆一句话区分

双塔是「先总结再比较」，ColBERT 是「先记笔记、比较时逐条对」，cross-encoder 是「把两篇放一起从头精读」。

两个延伸

  • ColPali（第 3、7 章）就是把 late interaction 原封搬到视觉上：页面截图 → VLM 编码成 patch 网格多向量，query token 对 patch 做 MaxSim——「多向量 + MaxSim」是可迁移模式，不绑定文本。
  • LLM-as-reranker（RankGPT 一类）是 cross-encoder 的当代变体：直接让 LLM 对候选做 pointwise 打分或 listwise 排序。更灵活（可加指令如「优先最新版本文档」），但延迟成本高一截，适合离线评估或低 QPS 高价值场景。

实践选择：默认 hybrid 召回 + cross-encoder 精排（§9 清单）；cross-encoder p99 扛不住或查询全是代码符号/型号时考虑 ColBERT；两阶段都上不起就先保证 hybrid + RRF。

§7查询变换：修 query 而不是修索引

前面各环节都在索引侧下功夫；查询变换在查询侧解决「用户的问题不是好的检索键」：

技术机制解决的失败模式
HyDE
arXiv:2212.10496		LLM 先生成「假设答案文档」，拿它去 embed 检索query-doc 语义鸿沟：疑问句和陈述句在向量空间里不够近，doc-doc 相似度更可靠
Multi-query一问改写成多问，并行检索取并集单一表述踩不中语料的用词分布
Decomposition复杂问题拆成子问题逐个检索多跳问题一次检索拿不全证据（图 RAG 之外的轻量解法）
Step-back
arXiv:2310.06117		先抽象成上位概念问题检索背景知识过于具体的问题命中不了讲原理的文档

实证提醒：ARAGOG 评测（arXiv:2404.01037）显示这些技巧的收益高度依赖数据集——每个都要在自己的评估集上验证后再上线，它们同时都会增加一次 LLM 调用的延迟。

§8评估：没有评估集，一切都是猜

评估分两层：检索质量（素材有没有拿对）和生成质量(答案有没有用对素材)。

检索指标

指标含义什么时候看它
recall@k前 k 条里覆盖了多少应命中文档最重要——答案素材没进上下文，后面全白搭。调 chunking/embedding/hybrid 主要看它
MRR第一条相关结果排名的倒数top-1 体验敏感的场景（如引用展示）
nDCG@k分级相关性 + 位置折扣调重排序时的标准指标

RAGAS 四指标（LLM-as-judge）

考检索的两个

context precision：相关块是否排在前面；context recall：ground truth 的信息被检索上下文覆盖的比例。

考生成的两个

faithfulness：答案与检索上下文的事实一致性（反幻觉核心指标）；answer relevancy：答非所问检测。

警告LLM-as-judge 的坑

win-rate 式评测存在长度偏置和位置偏置（可达 5–15 个百分点）。第 5 章会看到一个惨烈案例：LightRAG 用 win-rate 报告大胜，而第三方用标准 QA F1 复测时它在 HotpotQA 上只有 2.4 分。凡是 win-rate 数字，先问评测方式。

§9生产配置清单

把本章压缩成一张可以直接抄的默认配置表——每一项都可被你的评估集推翻：

环节起手默认升级触发条件
解析见第 3 章决策树—
Chunkingrecursive 512 token · 10% overlap · 标题元数据recall 低且失败案例是「语境断裂」→ contextual retrieval
Embedding中文选 Qwen3/BGE-M3，API 生态选 OpenAI v3自建 100 对评估集测 recall@10 后再换
索引pgvector HNSW（已有 PG）或 Qdrant>1000 万向量或静态大语料 → IVF/DiskANN
检索hybrid（dense + BM25）+ RRF k=60，召回 top-50几乎不需要改，这就是终点配置
重排BGE-reranker-v2-m3 或 Cohere，取 top-5延迟敏感 → ColBERT；预算紧 → 先砍这里以外的 LLM 调用

评估100–200 对人工评估集 + recall@k；上线后 RAGAS 抽样监控—

当这套管道仍解决不了的失败模式出现时——多跳关联（→ 第 4、5 章图方案）、超长专业文档（→ 第 6 章树方案）、图表扫描件（→ 第 3 章视觉路线）——才是升级架构的时机。
