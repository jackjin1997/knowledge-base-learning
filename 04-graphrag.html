微软 GraphRAG 精读：为全局问题付出的代价

「这个数据集的主要主题是什么？」——没有任何一个 chunk 包含这个答案，向量检索从原理上就不适用。GraphRAG 用实体图 + Leiden 社区 + 预生成摘要回答这类问题，本章沿代码走一遍它的全部机制与成本。

论文 arXiv:2404.16130 本地 ../graphrag 核心 Leiden + 社区报告 查询 Map-Reduce

§1它解决什么问题：QFS，不是检索

论文《From Local to Global》把矛头指向传统 RAG 的一个根本盲区：全局性问题（global questions）本质是查询聚焦式摘要（Query-Focused Summarization, QFS）任务，不是显式检索任务。「主要风险是什么」「有哪些反复出现的主题」这类问题的答案分散在整个语料中，top-k 相似度检索无论 k 取多少都拿不到。

GraphRAG 的解法是把「理解语料全局结构」的工作前移到索引期：

  1. LLM 遍历所有文本块，抽取实体、关系及其描述，构建实体图；
  2. Leiden 算法做层级社区检测，自底向上为每个社区预生成「社区报告」——独立于任何查询存在的多分辨率全局摘要。

论文报告：在约 100 万 token 语料上，对 sensemaking 类问题，答案全面性（comprehensiveness）和多样性（diversity）以 70–80% 胜率优于向量 RAG。注意这是 LLM-as-judge 的 win-rate（第 2 章讲过这类数字的局限）；独立复测中它在多跳 QA 上的 F1 是 38.5（MuSiQue），低于强稠密检索器的 45.7——GraphRAG 赢在全局综述，不赢在事实检索。

§2索引流水线：8 个 workflow

索引被编排为一串 workflow，注册在 pipeline factory（packages/graphrag/graphrag/index/workflows/factory.py:52-97），由 api/index.py:71-83 的 build_index → run_pipeline 逐个执行：

token 滑动窗口，默认 1200 token、overlap 100：graphrag-chunking/chunking/token_chunker.py:45-69，start_idx += chunk_size − overlap。无 LLM。
2
extract_graph — 实体/关系抽取 ⚠ 成本大头（≈75%）
每个 text_unit 一次 LLM 调用 + gleaning 补抽（§3）：index/operations/extract_graph/extract_graph.py:23-73
3
summarize_descriptions — 描述合并
同名实体在多个 chunk 中的描述先列表聚合，超 token 预算时 LLM 摘要：operations/summarize_descriptions/description_summary_extractor.py:89-117
4
finalize_graph — 计算度数、最终化节点与边
5
extract_covariates — claims 抽取（可选，默认关）
6
create_communities — Leiden 分层社区检测 ★ 核心资产
graphs/hierarchical_leiden.py:11-26，调 graspologic_native，max_cluster_size=10。图算法，无 LLM。
7
create_community_reports — 逐层生成社区报告 ⚠ 第二成本源
operations/summarize_communities/summarize_communities.py:61-73；上下文超预算时用子社区报告替换原始数据——分层递归压缩。
8
generate_text_embeddings — text_units / entities / reports 向量化
与第 2 章管道对照：GraphRAG 在「chunking 之后、embedding 之前」插入了 5 步图构建。这 5 步就是它的全部差异，也是全部成本来源。

实体抽取的输出格式

抽取 prompt（prompts/index/extract_graph.py:6-126）要求 LLM 输出分隔符格式的记录流，entity_types 可配置：

prompts/index/extract_graph.py（格式约定）("entity"<|>实体名<|>实体类型<|>描述)
##
("relationship"<|>源实体<|>目标实体<|>关系描述<|>强度)
##
...
<|COMPLETE|>

多个 chunk 抽到同一实体时的聚合逻辑是纯 pandas（无 LLM）——描述列表化留给步骤 3 处理，关系权重直接求和：

index/operations/extract_graph/extract_graph.py:104-130def _merge_entities(entity_dfs) -> pd.DataFrame:
    all_entities = pd.concat(entity_dfs, ignore_index=True)
    return (all_entities
        .groupby(["title", "type"], sort=False)
        .agg(description=("description", list),   # 描述列表化，待摘要
             text_unit_ids=("source_id", list),
             frequency=("source_id", "count")))

def _merge_relationships(relationship_dfs) -> pd.DataFrame:
    return (pd.concat(relationship_dfs)
        .groupby(["source", "target"], sort=False)
        .agg(description=("description", list),
             weight=("weight", "sum")))          # 权重求和

§3Gleaning：用成本换召回的多轮补抽

单次抽取会漏实体。GraphRAG 的解法是对话式补抽（gleaning）：先追加 CONTINUE_PROMPT（「再补抽，别漏」）拿增量结果，再用 LOOP_PROMPT 问模型「还有吗（Y/N）」决定是否继续，直到答 N 或达 max_gleanings（默认 2）：

index/operations/extract_graph/graph_extractor.py:99-122if self._max_gleanings > 0:
    for i in range(self._max_gleanings):
        messages_builder.add_user_message(CONTINUE_PROMPT)
        response = await self._model.completion_async(...)
        results += response.content
        if i >= self._max_gleanings - 1: break
        # 关键：问模型是否还有遗漏
        messages_builder.add_user_message(LOOP_PROMPT)
        response = await self._model.completion_async(...)
        if response.content != "Y": break   # 模型说完了，提前退出

max_gleanings=2 意味着每个 chunk 最多 5 次 LLM 调用（1 初始 + 2 补抽 + 2 判断）。这是「抽取占索引成本 75%」的直接原因——也是调优时第一个该动的旋钮。

§4Leiden 社区与分层报告

抽取出的实体图交给 Leiden 算法做层级社区检测——这一步是纯图算法，不花 LLM：

graphs/hierarchical_leiden.py:11-26def hierarchical_leiden(edges, max_cluster_size=10, random_seed=0xDEADBEEF):
    return gn.hierarchical_leiden(       # graspologic_native
        edges=edges, max_cluster_size=max_cluster_size,
        seed=random_seed, resolution=1.0,
        use_modularity=True, iterations=1)
    # 返回 (level, cluster_id, parent_id, [nodes]) 层级树

社区报告是结构化 JSON（prompt 在 prompts/index/community_report.py:5-153）：title / summary / rating(0-10) / findings[]，并强制数据引用格式 [Data: Entities (ids); Relationships (ids)]（单个引用最多 5 条 id，超出用 +more）+「不含引用的信息不写」的 grounding 规则——这套可溯源约定值得任何做摘要系统的人抄。

Global Search 的 Map-Reduce

这是 GraphRAG 查询侧最有辨识度的机制。Map 阶段对社区报告分批并行打分抽取要点（JSON 强制输出，score 0–100，「不知道」得 0 分），Reduce 阶段按分排序、截断到 max_data_tokens=8000 后合成最终答案：

query/structured_search/global_search/search.py:151-180（Map）+ 373-379（Reduce）# Map：并行让 LLM 评估各社区报告与问题的相关性
map_responses = await asyncio.gather(*[
    self._run_search(
        prompt=self.map_system_prompt.format(context_data=chunk, ...),
        query=query)
    for chunk in context_chunks])   # concurrent_coroutines=32

# 每个 map 结果形如 {"points": [{"description": "... [Data: Reports (2,7)]", "score": 85}]}

# Reduce：按得分降序、token 截断后合成 markdown 答案
sorted_by_score = sorted(map_responses, key=lambda x: x.get("score", 0), reverse=True)

Map/Reduce 的 system prompt 分别在 prompts/query/global_search_map_system_prompt.py:6-85 和 …reduce_system_prompt.py:6-81。Reduce prompt 有个细节：要求保留情态动词（shall/may/will）并抹去「analyst」角色痕迹。

§6成本画像与优化旋钮

公开实测数字

  • 官方 GitHub Discussion #440：索引约 120 万 token 语料 ≈ $8.2（GPT-4o）；单次 Global Search ≈ $0.4。
  • 学术实测：HotpotQA 全量语料索引消耗 7.57 亿 token，Naive RAG 只需 6200 万（约 12 倍）；GPT-4o 索引 HotpotQA 花费 $389。
  • 更新代价：新增文档改变社区划分需重建报告——LightRAG 论文估算 1399 个社区的重建 ≈ 1399 × 2 × 5000 token。

§7LazyGraphRAG：微软对自己的纠偏

2024.11，微软研究院发布 LazyGraphRAG，思路是把 LLM 的使用推迟到查询时：

  • 索引期零 LLM 调用，也不做预摘要：用 NLP 名词短语抽取建概念共现图，图统计量做社区划分——索引成本仅为 GraphRAG 的 0.1%，与向量 RAG 相当；
  • 查询期「先广度后深度」迭代加深：向量 + 图社区结构定位相关片段，小预算 LLM 做相关性判断与声明抽取；
  • 官方数据：答案质量与 Global Search 相当，查询成本低 700 倍以上。
结论这告诉我们什么

社区把 LazyGraphRAG 解读为微软对重索引路线的自我否定：图结构有价值，但用 LLM 预生成全部摘要不划算。这与下一章 LightRAG 的动机（砍掉社区摘要）殊途同归——两家从不同方向收敛到同一个判断。「索引期做多少、查询期做多少」是贯穿第 4–6 章的核心权衡轴：GraphRAG 全押索引期，PageIndex 全押查询期，LightRAG 和 LazyGraphRAG 在中间。
