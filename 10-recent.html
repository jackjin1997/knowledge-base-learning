2025–2026 编年：RAG 被吸收的两年

这两年发生的事比前五年加起来还多：deep research 一年走完「高端产品→免费→开源权重」，RL 教会模型自己检索，记忆系统从论文变成 API 一等公民，「context engineering」取代 RAG 成为学科名。本章按月份把这两年钉在时间线上——2026 上半年的内容来自专项实时调研。

算法 Search-R1 → Tongyi DR 叙事 RAG → Context Engineering 记忆 Zep / mem0 / MemOS 截至 2026-07

两年几十个事件，先给骨架——四条相互独立又互相成就的主线：

§3专题：RL 学检索谱系

这是 2025 年最重要的算法叙事：检索行为（何时搜、搜什么、搜几轮）从人写的工作流变成 RL 训练出的模型能力。谱系一年内完成从论文到可商用开源模型的闭环：

时间工作关键机制标志性数字
2025.03Search-R1只用最终答案奖励 + 检索 token masking 稳定训练Qwen2.5-7B 相对 RAG 基线 +41%
2025.03DeepRetrievalGRPO，直接以真实搜索引擎召回率为奖励优化查询改写查询改写不再靠 prompt 模板（对照第 2 章 §7）

2025.07WebSailor合成高不确定性任务 + DUPO（重复采样非零方差样本，rollout 效率 2–3×）7B 在 BrowseComp-en 6.7% 超 32B 前辈的 2.5%
2025.09Tongyi DeepResearch30.5B MoE（激活 3.3B），Apache 2.0 全开源HLE 32.9 / BrowseComp 43.4 / xbench 75.0——追平闭源
2026 H1方向转变：搜索深度控制AutoSearch（自适应搜索深度）、SAAS（抑制过度搜索）、CuSearch（课程式 rollout）问题从「学会搜」变成「别搜太多」——成本意识进入 RL 目标

评测也随之换代：BrowseComp 头部模型 2026 年中已达 88–90%（第三方 BenchLM 数据，接近饱和）；新基准转向「报告质量」——DeepResearch Bench II（9430 条专家 rubric，最强 agent 通过率不足 50%）、BrowseComp-Plus（ACL 2026，隔离检索器与 agent 的可复现评测）、ResearchRubrics。

§4专题：记忆系统两年演化

第 7 章 §6 介绍了 mem0/Letta/Graphiti 的机制，这里补上时间维度——记忆系统这两年走了三步：

1
2025 上半年：论文密集期
Zep 时序 KG（01）→ A-MEM 自组织笔记 + HippoRAG 2（02）→ mem0 生产级管道 + Letta sleep-time compute（04）→ MemOS「记忆操作系统」三类记忆统一调度（05）。核心共识形成：记忆 ≠ 只读检索，需要写入、整合、失效、后台维护。
2
2025 下半年–2026 初：平台化
Anthropic 把 memory tool / context editing / compaction 做成 API 一等公民（2026.04 Claude Managed Agents 持久记忆 public beta，记忆以文件形式可导出编辑）；腾讯 ima 把四类记忆模块（设定/画像/长期记忆/经验技能）落成 C 端产品功能。
3
2026 上半年：升维与反思并存
Letta 从记忆框架转型模型无关 agent harness（Letta Code）；Zep 定位升级「Context Lake」（Graphiti 2.7 万星、MCP Server 1.0）；mem0 新算法自报 LoCoMo 92.5；学术侧出现系统综述《From Storage to Experience》与批判论文《Contextual Agentic Memory is a Memo, Not True Memory》——检验期开始。Mem0 的《State of AI Agent Memory 2026》（2026.07.02）给出行业快照：情节/语义/程序三类记忆分型 + 六大未解难题（时间抽象、记忆陈旧性、隐私架构等）。
判断记忆公司集体「升维」说明了什么

mem0、Letta、Zep 在 2026 年不约而同从「记忆库/SDK」转向「harness / Context Lake / 平台」——单独的记忆层不构成产品护城河，记忆正在成为 agent 运行时的内置属性，就像 RAG 之于 Gemini File Search：被吸收进平台。给自建者的推论：除非有特殊合规/私有化需求，记忆层优先用平台能力而不是自己搭。

§6专题：Agent + Wiki——知识从「检索对象」变成「编译产物」

2026 年 4 月起你大概率在时间线上刷到过「wiki」。这波热度是两层叠加：

第一层铺垫（2025）：万物 wiki 化的产品线

产品时间做什么要点
DeepWiki（Cognition）2025.04URL 里 github.com 换成 deepwiki.com，即得仓库的 wiki 式文档 + 架构图 + 问答Devin Wiki 的免费公开版，索引 5 万+ 仓库；可用 .devin/wiki.json 引导页面结构（≤30 页）；2025.05 推官方 MCP server 供其他 agent 调用。闭源；开源复刻 DeepWiki-Open（17k stars）流程可查：克隆 → 代码切块 embedding → LLM 先定页面结构再逐页生成 → mermaid 画图，问答走 RAG
Zread.ai（智谱）2025.07「国产 DeepWiki」，GLM 底座中文语境优化，有 CLI 可本地生成
Grokipedia（xAI）2025.10AI 生成的百科：上线 88.5 万条目，2026.02 破 610 万（≈英文维基 79%）争议最大：部分条目近乎逐字 fork Wikipedia、用户不能直接编辑只能提建议由 AI 审核、被多方指出偏向与事实错误——「AI 批量生成百科」可行性与可信度的对照试验
Code Wiki（Google）2025.11公开仓库的持续自动维护 wiki主打与 DeepWiki 的差异点：代码变更后文档自动重建；Gemini 聊天以 wiki 为上下文答题

第二层导火索（2026.04）：Karpathy 的《llm-wiki》

2026 年 4 月 4 日 Karpathy 发布 gist《llm-wiki》，推文浏览量 1700 万+，gist 一周 5000+ stars、两上 HN 首页。他对 RAG 的批评一句话：「没有积累」——每次查询都对原始 chunk 重新理解，昨天综合出的认识今天全部丢弃。方案是让 agent 增量维护一个 Markdown wiki，把综合（synthesis）从查询时挪到写入时：

三层结构

raw/ 不可变原始资料；wiki/ LLM 全权维护的实体页/概念页/综述页，用 [[双链]] 互联，index.md 做全局目录；schema 文件（如 CLAUDE.md）定义命名规范与工作流。

三个操作

ingest：消化新资料，一次更新 10–15 个相关页面；query：从 index.md 导航展开作答，好答案回填为新页面；lint：周期体检——矛盾、过期论断、孤儿页、缺失交叉引用。

关键判断

约百篇来源的中等规模不需要任何 embedding 基础设施；wiki 之所以维护得下去，是因为 LLM 让维护成本趋近于零。「Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。」

社区 48 小时内跟进：Graphify（AST 本地解析 + SHA256 缓存 + git hook 自动更新，宣称单查询 token 降 71.5 倍——厂商自报）、Hermes Agent 内置 wiki Skill、OpenClaw memory-wiki、跨 CLI 的 llm-wiki-agent。

放进本教程的坐标系里看

  • 思想不新，形态新：「预生成结构化中间表示、拿索引期算力换查询期质量」正是 GraphRAG 社区报告（第 4 章）和 RAPTOR 摘要树（第 6 章）的思想——wiki 是它的人类可读、agent 可编辑、git 可版本化版本。
  • 检索方式是导航不是相似度：index.md 是入口、双链是图的边，agent 像人读维基一样逐跳展开——天然多跳，且全程可 grep、可 git diff、可人工审查。这与 PageIndex 树上推理、Claude Code grep 检索同属「结构 + 推理」一族。
  • 与记忆系统合流：A-MEM（2025.02）早已提出 Zettelkasten 式互联笔记做 agent 记忆；Karpathy 版把它平民化成「一个文件夹 + 三个 prompt」。Wiki 从给人看的文档，变成 agent 的可读写长期记忆格式——这是「wiki 复兴」的真正内核——它与做梦/睡眠计算如何合流成「后台认知层」，见第 11 章。
  • 适用边界（社区共识）：个人/小团队、10 万 token 级语料优于 RAG；大规模、高频变更、多用户并发语料仍属 RAG 与托管检索。反方论点也值得记录：HN 有人称其为「多几步的 RAG」，并援引模型自噬研究（Shumailov et al., Nature 2024）质疑 LLM 反复改写自产内容的漂移风险——lint 操作正是对这个风险的工程回应。
动手低成本试法

这套模式不需要任何新工具：在本 workspace 建一个 wiki/ 目录，让 Claude Code 读完一个仓库后按「实体页 + index.md + 双链」写笔记，下次提问先让它查自己的 wiki——你会直观感受到「编译过的知识」和「现场检索原文」的差别。事实上你正在读的这套教程，本身就是一次 ingest + 综合的产物。

§7站在 2026.07 的判断

观察对从业者的含义
托管化不可逆：Gemini File Search / OpenAI file_search 把「分块-嵌入-检索-引用」整条管道做成了 API通用场景自建管道的理由只剩：私有化合规、特殊语料（第 2 章的手艺仍是调优和评估这些托管服务的基础）

记忆进入检验期：产品升维平台，学术出现批判声音对记忆系统的宣称保持第 5 章式的怀疑：看独立基准，不看厂商自报
多模态检索成熟：embedding + reranker 成对的多模态方案 + 光学压缩持续迭代视觉密集语料（第 3 章 §8 的 ColPali 路线）从「前沿实验」变成「可选默认」
知识开始被「编译」：DeepWiki/Code Wiki 批量 wiki 化仓库，Karpathy 的 llm-wiki 让 agent 为自己维护知识库（§6）「写入时综合」与「查询时组装」成为一对新的架构选择；中小规模知识库出现了零向量库的可行路径
收束两年的一句话总结

2025 年初你还在问「怎么把 RAG 做好」，2026 年中正确的问题已经是「我的 agent 的上下文由哪些来源组成、谁来决定、成本谁付」——检索是其中一个来源，记忆是另一个，长上下文是装它们的容器，RL 是学会调度它们的方法。本教程第 2–6 章的机制知识没有过时：它们从「系统的全部」变成了「工具箱里必须懂的工具」。
