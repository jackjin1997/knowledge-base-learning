Chapter 07 · Frontier

前沿方案：自省、多跳、视觉与记忆

前六章覆盖了管道、图、树三大主干。本章补齐 2024–2026 的其余前沿：让检索会自我纠错、让多跳变成一次图算法、让检索跳过文本、以及 RAG 正在演化成的东西——记忆系统。

自省 Self-RAG / CRAG 多跳 HippoRAG PPR 视觉 ColPali 记忆 mem0 / Letta / Graphiti

§1自省式检索：Self-RAG 与 CRAG

Naive RAG 有两个盲目：盲目检索（不管问题需不需要）和盲目信任（检回什么用什么）。两个工作分别治这两个病：

Self-RAG arXiv:2310.11511

训练模型输出反思 token：[Retrieve] 要不要检索、[IsRel] 检回的相关吗、[IsSup] 生成有据吗、[IsUse] 答案有用吗。检索从固定步骤变成模型的自主决策——「按需检索」开山之作。代价：需要专门训练。

CRAG arXiv:2401.15884

免训练路线：轻量检索评估器把结果打为 Correct / Ambiguous / Incorrect 三档。正确→知识精炼（剔除无关句）；错误→丢弃并回退 web 搜索；模糊→两者结合。即插即用，可套在任何现有管道外。

今天这两个工作的具体实现已不常直接部署，但它们确立的模式无处不在：现代 agent 在工具调用后判断「这个结果够不够、要不要换个查询再来」，就是 Self-RAG/CRAG 思想在提示工程层面的复活。

§2HippoRAG：把多跳变成一次 PageRank

多跳问题（"A 公司的供应商的 CEO 是谁"）的传统解法是迭代检索——查一跳、读结果、再查一跳，每跳一次 LLM 调用。HippoRAG（OSU，NeurIPS 2024）的洞察来自海马体索引理论：人脑不迭代——新皮层存记忆内容，海马体存稀疏关联索引，回忆是索引上的一次模式补全。

HippoRAG 2（arXiv:2502.14802，"From RAG to Memory"）三处升级：段落节点整合进图（稠密-稀疏结合）、query-to-triple 匹配取代 NER（种子更准）、LLM「识别记忆」过滤检回三元组去噪。结果：MuSiQue F1 44.8→51.9、2Wiki Recall@5 90.4%，且简单 QA 不掉分——这是多数图 RAG 做不到的（结构化方法在单跳问题上通常掉 5–10 个 F1）。

§3图 RAG 生态与最硬的一张数据表

第 4、5 章讲了两个代表，生态里还有几个定位各异的玩家：

项目一句话定位
nano-graphrag微软 GraphRAG 的 ~1100 行极简实现，LightRAG 的代码底座，教学与魔改首选
KAG（蚂蚁）专业领域「严谨型」：schema 约束建模 + 图文互索引 + 逻辑形式引导推理，政务/医疗落地。HotpotQA F1 +19.6% arXiv:2409.13731
Graphiti / Zep不做文档 RAG，做 agent 记忆层：双时间轴时序 KG（§6 详述）
CogneeECL（Extract-Cognify-Load）管道的「AI 记忆引擎」，图+向量混合，介于 RAG 框架和记忆之间
Youtu-GraphRAG（腾讯，ICLR 2026）schema 驱动的垂直统一 agent，换领域只改 schema arXiv:2508.19855

三个解读：① 强稠密检索器是硬基线——多数图 RAG 在通用 QA 上打不过它；② HippoRAG 系在关联最密的 2Wiki 上优势最大——图的价值集中在多跳；③ LightRAG 的崩盘数字与其他论文（GRASP 里 HotpotQA F1 0.819）直接矛盾——回看第 5 章 §7 的评测争议。另注意延迟：GRASP 实测图 RAG 查询普遍 30–50 秒（LightRAG 39.7–51.3s），向量 RAG 是亚秒级。

共识社区的务实结论

先用向量 RAG + rerank 打底（第 2 章清单），确认失败模式确实是「关系推理 / 全局综合」再上图；schema 密集领域（法律/医药/金融合规）图的收益最确定；混合架构（HybridRAG）通常优于单一方案。

§4视觉检索：ColPali 一族

第 3 章讲了怎么把 PDF 解析好，ColPali（arXiv:2407.01449）的回答是：别解析了。把整页文档图像交给 VLM 编码成 patch 网格，每个 patch 投影为 128 维向量，检索时用 ColBERT 式 late interaction（query token 对 patch 的 MaxSim 求和）打分——OCR、版面分析、chunking 三步全部消失。

  • 数字：ViDoRe 基准 NDCG@5 0.81 vs 传统「解析→切分→嵌入」管线 0.66；表格、图表、扫描件场景优势最大。
  • 谱系：ColPali（PaliGemma）→ ColQwen2.5 → ColSmolVLM；基准演进到 ViDoRe V3。
  • 代价：每页多向量存储开销大；检回的是页面图片，下游必须是多模态 LLM，且难做细粒度文本引用。
  • 实践折中：ColPali 做页面级召回 + VLM 对召回页即时解析——视觉检索管召回，文本管消费。

同一根线上的另一端是 DeepSeek-OCR 的「上下文光学压缩」（第 3 章 §3）：一个视觉 token 顶 10 个文本 token 而 OCR 精度保持 97%——文档图像可能成为长上下文与记忆的压缩载体。视觉不只是检索模态，还可能是存储格式。

§5Agentic RAG 与 Deep Research

把前面所有技术收进一个更大的转变：检索从「流水线的固定第一步」变成「agent 循环中的工具」。模型在推理中发现知识缺口 → 发起定向查询 → 读结果 → 决定继续查还是回答（ReAct、Self-Ask、Search-o1 一脉；综述 arXiv:2506.10408 把它分为 System 1 快速单跳与 System 2 规划/批评/验证两类）。

三个已在生产验证的形态：

Deep Research

多轮迭代检索 + 交叉验证 + 引用合成，一次任务几十次检索调用。OpenAI/Anthropic/Google 均产品化。

Agentic search（grep 式）

Claude Code 对代码库不建向量索引，用 grep/glob 当检索工具，靠模型多轮缩小范围——结构清晰的语料，遍历 + 推理胜过 embedding。

推理式树/图搜索

PageIndex 树上推理（第 6 章）、MiroFish 的 ReAct 多工具报告（第 8 章）——检索工具本身也可以是结构化索引。

视角Agentic 不是免费的

每轮工具调用都是一次 LLM 前向：延迟和成本随轮数线性涨。经验分界——高价值低频查询（尽调、研究、审计）用 agentic 多轮；低价值高频查询（客服 FAQ）用第 2 章的单发管道。中间地带才是路由/自省技术（§1）的用武之地。

§6记忆系统：RAG 的下一站

RAG 是只读的：语料不因对话而变。Agent 记忆系统补上写路径——有写入、压缩、遗忘策略的知识库。这是「From RAG to Memory」（HippoRAG 2 的副标题）所指的方向：

系统机制与 RAG 的关键差异
mem0agent 外置记忆 CRUD 层：向量+图+KV，自动抽取「值得记住的事实」有写入与更新决策（记什么、改什么）
Letta（MemGPT）记忆分层 core / recall / archival，放进 agent 循环由模型自管——类比 OS 内存分页模型自己管理记忆的进出
Graphiti / Zep双时间轴时序 KG：每条边带「事实有效期」+「摄入时间」，新信息不删旧边而是标记失效支持「某时刻什么为真」的时间点查询；DMR 94.8%，检索融合语义+BM25+图遍历且查询路径无 LLM

时序这条线值得多看一眼：知识会过期（「A 是 B 公司 CEO」在 2024 为真、2026 为假），静态 RAG 只能覆盖重写，时序 KG 保留完整演化史。本地项目 MiroFish 正是用 Zep/Graphiti 做 agent 群体的「集体记忆」，模拟中的舆论反转依赖事实失效机制——第 8 章展开。记忆系统「谁来维护」的问题（睡眠计算、dreaming、wiki lint）在第 11 章成体系展开。

§7框架生态速览

框架定位选它当
LlamaIndex检索/索引专精的开发框架需要细粒度控制索引结构、检索器组合
Haystack（deepset）组件化管道、企业级部署生产管道的工程化、可观测性优先
RAGFlow开箱即用 RAG 引擎，强项是 DeepDoc 深度文档解析 + 低代码 UI要快速上线且文档质量复杂
DifyLLM 应用平台，RAG 只是其 workflow/agent 能力之一团队要的是应用平台不是检索库

与自研的关系：框架解决的是编排和集成，本系列讲的机制（chunking 策略、混合检索、重排、图/树索引）在任何框架里都要自己做决策——这正是读完第 2–6 章的价值。
