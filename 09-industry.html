Chapter 09 · Industry Case Studies

工业界产品拆解：Manus、Kimi 与豆包的三条检索路线

论文的数字可以互相矛盾（第 5 章见识过了），但产品的取舍被真实流量验证过。本章拆三个成熟产品：Manus 押注上下文工程、Kimi 押注长上下文 + 端到端 RL 的自主搜索、豆包押注搜索栈 + 托管知识库——并解剖用户直觉「豆包法条做得不错」背后的机制。

Manus KV-cache 优先 Kimi agentic RL 搜索 豆包 搜索栈 + VikingDB 对照 法律垂直 RAG

§1为什么看产品而不是只看论文

三个理由：① 图 RAG 领域的论文评测已被证明高度不可靠（同一方法跨论文 F1 差 40 倍，第 5 章 §7）；② 产品的架构取舍要同时满足延迟、成本、规模、合规——这些约束论文里没有；③ 三家恰好代表了第 1 章五族之外的三种「组合策略」，且都有可溯源的公开材料。需要先声明口径：本章事实均标注来源；产品未公开的部分明确标为推断，尤其是豆包——它几乎没有官方技术披露，解剖主要靠火山引擎公开文档 + 合理推断。

§2Manus：把检索问题变成上下文工程问题

一手来源是 Manus 联合创始人 Peak Ji 的官方博客《Context Engineering for AI Agents》——可能是 2025 年被引用最多的 agent 工程文章。背景：Manus 决定不自训模型、全部押注上下文工程，agent 框架重写了四次。六条经验，每条都与「检索/记忆」直接相关：

原则机制关键数字/细节
① KV-cache 命中率是第一指标agent 是追加式循环，输入:输出 ≈ 100:1，前缀稳定才能命中缓存Claude 缓存输入 $0.30/MTok vs 未缓存 $3.00——10 倍差价。系统提示里放一个精确到秒的时间戳就毁掉全部缓存
② Mask 而非移除工具动态增删工具会毁缓存 + 让历史步骤引用失效；改为解码期用 response prefill 约束可选工具统一命名前缀（browser_ / shell_）实现按类批量约束。他们实验过 RAG 式动态加载工具定义，因破坏缓存而放弃
③ 文件系统 = 终极上下文把文件系统当无限大、持久、可操作的外置记忆；压缩必须可恢复——丢网页正文但留 URL、丢文档内容但留路径128K 窗口在真实任务中不够：一条网页观测就可能爆掉
④ Recitation 复述持续重写 todo.md 并逐项勾选，把全局计划反复推到上下文末尾（近期注意力区）典型任务约 50 次工具调用，复述治目标漂移与 lost-in-the-middle
⑤ 保留错误失败动作和堆栈留在上下文里，模型隐式更新先验「错误恢复是真正 agent 行为的最清晰指标」——而学术基准只测理想条件
⑥ 反 few-shot 套牢重复的 action-observation 对诱导机械模仿，需受控注入结构化变异案例：批量审 20 份简历时陷入重复漂移

它怎么做「检索」

公开叙事中 Manus 没有向量 RAG 组件（推断：无逐字官方否认，但所有公开材料一致指向文件系统路线）。其动作空间分三层：第一层 ≤20 个稳定原子工具（bash、文件读写、搜索、浏览器）；第二层通过 bash 调沙箱内 CLI——检索靠 grep 而非专用检索工具；第三层是代码。超出单上下文的规模问题交给 Wide Research（2025.07）：主 agent 分解任务 → 100+ 个完整 Manus 实例并行（各有独立 VM 与上下文，子 agent 间不通信以防上下文污染）→ 主 agent 汇总。

启示与传统 RAG 的关系

Manus 把召回从「离线索引问题」改写为「在线决策问题」：检索单位从 chunk 变成文件 + 路径 + 工具，URL/路径是指针、内容随取随弃。它优化的不是「塞进多少相关内容」而是 KV-cache 命中率。前提是「前沿模型 + 沙箱执行环境」——对无沙箱的纯问答知识库，向量 RAG 仍更经济（推断）。todo.md 复述、保留错误、mask-not-remove、可恢复压缩四个技法可直接迁移到任何 RAG/agent 系统。

时效备注：Meta 于 2025.12 宣布收购 Manus（WSJ 称超 20 亿美元），2026.04 中国发改委要求撤销交易、Meta 正在 unwind——引用商业信息请注明时间点。

§3Kimi：长上下文原教旨 + 端到端 RL 的自主搜索

「蜜蜂模型」：对 RAG 最锋利的批评

2023.10 Kimi Chat 以 20 万字上下文发布时，杨植麟把业界长文本「捷径」分为三类并逐一否定：金鱼（滑动窗口，健忘）、蜜蜂（降采样 / RAG 检索增强——只见局部不见整体）、蝌蚪（缩参数换长度）。RAG 被明确归入「蜜蜂」类有损捷径。2024 年初的专访里他的立场浓缩为一句「lossless long context is everything」。

溯源那句网传名言的核查结果

流传甚广的「RAG 是妥协、long context 才是终局」没有逐字出处——专访全文里找不到这句话。它是社区对两处真实表态的概括：① 2023.10 把 RAG 归入「蜜蜂」式有损捷径；② 2024 专访的「lossless long context is everything」与「如果你有 10 亿 context length，今天看到的问题都不是问题」。引用时请用真实表述 + 上述两个出处。同样注意：「200 万字无损」是产品宣传口径，具体网络结构未公开。

公开的技术支柱有两个：MoBA（arXiv:2502.13189，把 MoE 思想引入注意力——上下文切块、可训练 gating 让每个 query 自选关注块，论文确认已部署支撑 Kimi 长上下文）；Mooncake（arXiv:2407.00079，FAST'25）——以 KV-cache 为中心的 prefill/decode 分离推理架构，用集群闲置 CPU/DRAM/SSD 建全局缓存池「以存换算」，同等硬件多承接约 110% 请求。长上下文路线能走通，一半靠注意力结构，一半靠让它算得起的基建——这与 Manus 把 KV-cache 当第一指标殊途同归。

检索没有消失，而是被 RL 内化

有趣的是：嘴上说 RAG 是蜜蜂，Kimi 的产品线却把「检索」越做越重——只是形态完全不同：

1
基础联网搜索
检索到的整页网页塞进长上下文精读（而非取片段）——长上下文与检索互补的最直接产品化。
2
探索版（2024.10）：agentic search
自主规划搜索策略、搜索量 10 倍、一次精读 500+ 页面、对结果反思后多轮补搜。
3
Kimi-Researcher（2025.06）：端到端 agentic RL
不是工作流编排——整个「搜索→浏览→写代码→合成」行为链用 REINFORCE 端到端训练：严格 on-policy、负例部分丢弃防熵坍缩、γ 衰减奖励激励短路径。单任务平均 23 个推理步、探索 200+ URL、仅 3.2% 内容进最终报告。HLE Pass@1 从 8.6% → 26.9%，官方称提升「几乎完全来自端到端 RL」。
4
K2 Thinking（开源）
边思考边调工具（interleaved thinking + function calls），可连续 200–300 次工具调用不漂移；BrowseComp 60.2% vs 人类基线 29.2%。Researcher 的 RL 经验沉淀进了开源基座（推断）。
启示Kimi 路线的边界

检索从「pipeline 的固定预处理」变成「RL 习得的自主行为」：长上下文管读得全，agentic search 管找得到。传统 RAG 的不可替代性收缩到三类场景：私有/权限敏感语料（公网搜索够不到）、规模远超窗口且全量入 context 不经济、高频重复查询的成本控制——正好呼应第 7 章 §5 的「高价值低频 vs 低价值高频」分界。

§4豆包：法条场景解剖

用户直觉「豆包对法条做得不错」是个好的解剖入口。先说结论：没有任何官方证据表明豆包内置了结构化法条索引。可查证的机制是三层：

① 参数化记忆（推断）

法条是高频、稳定、结构规整的文本，中文预训练语料覆盖充分——模型「背得住」。旁证：公开的失败案例集中在事实性/程序性细节编造（如退票手续费规则答错致损引诉讼），而非法条编号错引。

② 字节搜索栈联网检索（事实+推断）

豆包联网检索标注引用来源；法条查询可命中国家法律法规数据库、北大法宝等权威站点直接返回原文。法条是抖音/头条搜索长期优化的高频类目，权威站点加权是搜索栈的存量能力（推断）。

③ 深度思考核验（事实）

官方披露新版深度思考「边想边搜」：推理中多次调用搜索、自我质疑、假设验证与错误修正——对法条编号、修订版本做二次核验。

法律行业侧唯一可查的官方案例是火山引擎 × 威科（Wolters Kluwer）：豆包大模型对判例做批量字段抽取（法条、金额、地点等上百字段，官方称精度 95%+）服务威科先行法律库——说明字节在法律 NLP 结构化上有工程积累，但这是 B 端信息库建设，不等于 C 端法条问答的实现。

§5对照：法律垂直 RAG 为什么必须结构化

豆包靠「记忆 + 搜索」覆盖大众法条问答；但专业法律产品全部走了另一条路——结构化法条库 + 可验证检索。原因在法条这种语料的三个特点，恰好都是纯向量检索的盲区：

三个走结构化路线的对照产品：通义法睿（阿里：千问基座 + 法律精调 + RAG + 司法专属小模型）；ChatLaw（北大：Keyword LLM 从模糊提问抽关键词 + 专用 Law Embedding，93 万判例语料）；DISC-LawLLM（复旦，DASFAA 2024：法律三段论 prompting 构造 SFT 数据 + 可验证检索模块，LawBench 仅次于 GPT-4）。

合流这一节与全书的连接

法条库是「树族方案」的完美应用场景：编-章-节-条是现成的 PageIndex 式目录树（第 6 章），修订时间线对应 Graphiti 式时序失效机制（第 7 章 §6），效力层级/地域是向量库的标量过滤（第 2 章 §4）。如果你要做法律知识库，这三章的组合就是蓝图——豆包的「记忆 + 搜索」路线只适合大众咨询，专业场景必须结构化。

收束三家共同证明的一件事

没有一家在做 2023 年教科书式的「chunk + embedding + top-k」——但也没有一家真正抛弃检索。Manus 把它变成文件系统工具，Kimi 把它变成 RL 行为，豆包把它藏进搜索栈和托管管道。检索作为能力永生，RAG 作为固定架构已经溶解——这就是第 1 章「From RAG to Context」判断在产品端的三重验证。而 KV-cache 在 Manus（第一指标）和 Kimi（Mooncake）两家同时成为成本核心，提示上下文工程时代的新公共底座。
