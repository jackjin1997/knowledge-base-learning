本地实战与选型：两个生产系统，一张决策图

前面的章节讲了「学术界怎么设计」，本章看 workspace 里的真实系统怎么取舍：gbrain 用零 LLM 的图替代 NER，MiroFish 把时序图谱当 agent 群体的记忆，RAGFlow 把全书技术集成为一个产品。最后把仓库放到同一根轴上，给出选型指南。

§1gbrain：把「图」做成零 LLM 的确定性系统

gbrain（TypeScript/Bun，PGLite 或 Postgres+pgvector）是面向代码库与团队知识库的检索系统。它最有教学价值的决策是：同样要「图」，它拒绝了 LightRAG/GraphRAG 的 LLM 抽取路线——

零 LLM 自动链接

代码知识库里的实体关系大多是显式的：markdown 链接、wiki 双链、路径引用、符号名。gbrain 用三套正则 + 启发式类型推断直接抽（src/core/link-extraction.ts:extractPageLinks()），零 LLM 调用、零抽取幻觉、结果可复现。对照第 7 章的批评「图 RAG 抽取错误率 15–20% 且错误会传播」——当关系本来就写在文本里时，不需要让 LLM 猜。

§2MiroFish：知识图谱作为 Agent 群体的集体记忆

MiroFish（Python/Flask + Vue3，AGPL-3.0）不是检索系统——它是社会模拟引擎：现实种子文本 → 知识图谱 → 数千个带人格与记忆的 agent 在类 Twitter/Reddit 双平台里模拟 → 预测报告。它在本系列的价值：展示知识图谱除「检索」外的另一种消费方式——作为多智能体系统的共享记忆基底。五阶段流水线中与知识库相关的三个设计：

① Ontology-First：先定制 schema 再抽取

不用通用实体抽取。第一步让 LLM 分析语料 + 模拟需求，生成为本次任务定制的本体（entity_types / edge_types 的 JSON 定义，services/ontology_generator.py），再把这个 schema 交给抽取引擎。对照第 7 章的 KAG 与 Youtu-GraphRAG——「schema 约束抽取」是图质量的最有效杠杆，MiroFish 把 schema 本身也交给 LLM 按需生成。

② 时序知识图谱：事实有生命周期

图构建托管给 Zep Cloud（底层 Graphiti，第 7 章 §6），每条边携带完整时间维度（services/graph_builder.py:74-82）：

backend/app/services/graph_builder.py:74-82（边数据结构）edges_data.append({
    "fact": edge.fact,                # 事实文本
    "created_at": ...,   # 何时被观察到
    "valid_at": ...,     # 从何时开始为真
    "invalid_at": ...,   # 从何时失效（不删除，标记）
    "expired_at": ...,
    "episodes": episodes, # 溯源：来自哪些原文片段
})

模拟运行中 agent 的行为可以写回图谱（zep_graph_memory_updater.py）——舆论反转 = 旧事实被标记失效 + 新事实生效。静态 RAG 无法表达这种演化。

③ ReAct 多工具报告生成

最终报告不是「一次检索 + 汇总」，而是 agent 拿四个工具做多轮推理（services/report_agent.py + zep_tools.py，每章 ≤5 次工具调用 + 3 次反思）：quick_search（Zep 混合检索 + cross-encoder 重排）、panorama_search（可含已失效历史事实）、insight_forge（拆子问题→多轮检索→聚合）、interview_agents（直接采访模拟中的活体 agent）。这是第 7 章 agentic RAG 的完整生产实例。

局限作为案例的另一面

强依赖 Zep Cloud（闭源托管、配额、数据出境——社区有换 Graphiti+Neo4j 自托管的分支）；成本高（README 明确警告先跑 <40 轮）；Zep 语义检索缺少 GraphRAG 式全局摘要能力，全局判断靠 agent 群体涌现来补。

§3RAGFlow：把全书技术装进一个产品（../ragflow）

RAGFlow（infiniflow，2026-07 主分支已克隆到本 workspace）与前两个项目不同：它不是某一族方案的代表，而是把本教程各章技术做成一个开箱即用产品的工程集成商。用它来检验全书的体系最合适——每一章都能在它的目录树里找到对应模块：

教程章节RAGFlow 对应模块工程口径的差异
第 2 章 全链路rag/nlp/ 混合检索 + 多供应商 embedding/rerank融合不是 RRF 而是加权求和（weighted_sum, weights=0.05,0.95，rag/nlp/search.py:210）——默认向量占 95%，可配置但需调优；另有向量空结果时降 min_match 重试的降级策略
第 3 章 解析deepdoc/：YOLOv10 布局（10 类）+ TSR 表格结构（5 标签→HTML）+ 表格自动旋转管线派自研为主，2025.10 起同时集成 MinerU/Docling 作可选后端——与第 3 章「两级路由」思路一致
第 4、5 章 图族rag/graphrag/ 双轨：light/（注明源自微软 GraphRAG，带 gleaning）+ general/（Leiden 社区 + 社区报告）一个产品同时装了第 4、5 章两条路线，按知识库配置切换
第 6 章 树族rag/svr/task_executor_refactor/raptor_service.pyRAPTOR 支持 file / dataset 两级（:134-141）——跨文件聚合树是教程未覆盖的扩展
第 7 章 agentic / 记忆agent/ Canvas DSL 25+ 组件（2026 新增 Browser 自动化组件）；memory/ 初代记忆模块memory 消息带 valid_at / invalid_at / forget_at 字段（messages.py:34-120）——时序 + 遗忘的雏形，呼应第 7 章 Graphiti 的事实生命周期
第 10 章 MCP 化mcp/server/：知识库经 SSE/HTTP 暴露为 MCP 工具，带 LRU+TTL 元数据缓存「检索变成 agent 工具」的产品落地——外部 agent（如 Claude）直接调它检索

它真正的新东西

  • 模板化 chunking（rag/app/ 12 种）：naive/paper/laws/manual/book/resume/qa/table 各一个专用切分器——法律按条款序号切、手册按目录层级切、简历走 YOLOv10 布局 + 并行 LLM 抽 30+ 字段（resume.py:92-150）。这把第 2 章「document-based chunking」推到极致：切分策略跟着文档类型走，而不是一套参数打天下。
  • Entity Pool 查询改写（rag/graphrag/search.py:46-54）：改写 query 前先从图里采样已有实体类型作 TYPE_POOL 注入 prompt——用图的既有词汇约束 LLM，降低改写幻觉。第 2 章查询变换 + 第 5 章图检索的一个精巧组合。
  • 索引期 LLM 增强：Auto-keyword / Auto-question 为每个 chunk 生成关键词与「它能回答什么问题」——与 Anthropic contextual retrieval 同思路的另一种实现（问题比语境前缀更贴近查询分布）。
  • Python→Go 迁移（internal/）：解析热路径用 Go + CGO 绑定 pdfium 等原生库重写，Python 保留业务编排——生产 RAG 引擎的性能演化方向。
定位什么时候选它

要「两周内上线一个带 UI、带权限、文档类型杂」的企业知识库时，RAGFlow 这类一体化引擎是合理起点——本教程教你的是打开它每个配置项时知道自己在调什么（融合权重、chunk 模板、GraphRAG 开关、RAPTOR 层级）。反过来，需要极致定制（gbrain 式领域特化）或研究新范式时，一体化引擎的抽象反而碍事。

洞察一条贯穿五个项目的规律

「结构的确定性」与「LLM 参与度」成反比，与可靠性成正比。gbrain 的关系是文本里写死的 → 正则抽，零幻觉；PageIndex 的结构是文档自带的 → LLM 只做解析与校验；LightRAG/GraphRAG 的关系是隐含的 → 只能 LLM 猜，于是有 15–20% 抽取错误率和评测争议。选方案前先问：我的语料里，结构是显式的还是隐含的？显式就别用 LLM 抽。

§5选型决策指南

§6动手路线图

把学习转成肌肉记忆的四步，都用本 workspace 的仓库：

1
跑通对照实验（半天）
LightRAG 同一批文档跑 naive vs hybrid vs mix 三种 mode，观察上下文差异——一个仓库内置了向量/图两条路线的 A/B 环境。
2
感受成本结构（半天）
graphrag 对 50 页文档跑 graphrag index，看 cache 目录里每步的 LLM 调用量；把 max_gleanings 从 2 改 0 重跑对比抽取质量——第 4 章的成本表会变成体感。
3
树搜索直觉（2 小时）
PageIndex 对一份年报建树，读 examples/agentic_vectorless_rag_demo.py 的推理轨迹，看它每一步为什么进那个分支。
4
读一个生产级实现（1 天）
gbrain 的 src/core/search/hybrid.ts 从入口读到 RRF 融合——四路检索怎么在一个 SQL 数据库上落地、缓存怎么失效、成本模式怎么路由。
收尾回到第 1 章的判断

检索没有死，它正在变成 agent 上下文工程的工具箱：向量、BM25、图遍历、树推理、视觉召回——都是工具，语料形态和失败模式决定用哪把。本系列给你的不是「最佳方案」，而是判断力。
