RAG 演进史：从端到端模型到 Agent 的工具

「RAG」这个词在六年里指代过三种完全不同的东西。理解这三次漂移，比背诵任何一种架构都更能帮你判断下一步往哪走。

2020 Lewis et al. 2022 Naive RAG 2024 GraphRAG 2025+ Agentic RAG

§1「RAG」的三次定义漂移

先把一个常见误会拆掉：今天工程界说的 RAG，和 2020 年论文里的 RAG，不是同一个东西。这个词经历了三次语义漂移：

核心为什么这个区分重要

你在面试或论文里读到的「RAG 提升了 XX 指标」，必须先看清它说的是哪一代 RAG。原始 RAG（Lewis et al., arXiv:2005.11401）是把 DPR 检索器和 BART 生成器联合微调的模型架构，检索到的文档被当作隐变量做边际化（RAG-Sequence / RAG-Token 两个变体）。而今天你写的 RAG 系统里没有任何参数被训练——它是围绕冻结 LLM 的工程管道。两者共享的只有一个思想：参数记不住的知识，用检索补。

§2完整时间线

2017
DrQA（Facebook）
开放域问答的「检索 + 阅读器」两段式：TF-IDF/BM25 检索维基百科，阅读模型抽取答案。今天 RAG 架构的直系祖先。
2020.02
REALM（Google）
首次把检索器放进语言模型预训练，端到端联合训练——检索器由下游任务信号驱动学习。
2020.04
DPR（Karpukhin et al.）
双塔（dual-encoder）稠密检索 + 对比学习，首次系统性证明稠密向量可击败 BM25。今天所有「embedding + 向量库」范式的奠基工作。
2020.05
RAG 原始论文（Lewis et al., NeurIPS 2020）
「RAG」一词诞生。DPR + BART 联合微调，三个开放域 QA 达到 SOTA。arXiv:2005.11401
2021
FiD（Izacard & Grave）
Fusion-in-Decoder：每个检索段落独立编码、在 decoder 中融合，能吃下 100 个段落，长期霸榜 QA。
2021.12
RETRO（DeepMind）
从 2 万亿 token 数据库检索，chunked cross-attention 注入邻近块，以 25 倍更少参数达到 GPT-3 级性能——证明外部知识可以替代参数规模。arXiv:2112.04426
2022.11
ChatGPT 引爆 + LangChain / LlamaIndex
「给 LLM 接私有数据」成为第一大工程问题。Naive RAG 流水线定型；Pinecone/Weaviate/Chroma 向量库融资潮。
2022.12
HyDE
用 LLM 生成「假设答案文档」再检索，缓解 query-doc 语义鸿沟。查询变换技术起点。arXiv:2212.10496
2023.10
Self-RAG
模型输出反思 token 自主决定是否检索、自评生成质量——「按需检索」开山之作。arXiv:2310.11511
2024.01
RAPTOR / CRAG
RAPTOR 用递归聚类摘要构建多层树；CRAG 给检索结果加质量评估与纠错回退。「高级 RAG」技术密集涌现。
2024.04
Microsoft GraphRAG
「From Local to Global」：实体知识图谱 + Leiden 社区分层摘要，专攻全局性问题（QFS）。→ 第 4 章精读arXiv:2404.16130
2024.05
HippoRAG（NeurIPS 2024）
海马体记忆理论启发：OpenIE 三元组图 + Personalized PageRank，一次图遍历完成多跳。arXiv:2405.14831
2024.07
ColPali
跳过 OCR 和解析，把 PDF 页面截图直接编码为 patch 多向量做 late interaction 检索——视觉检索路线开端。arXiv:2407.01449
2024.09
Anthropic Contextual Retrieval / Jina Late Chunking
两条路线解决同一个问题——chunk 失去全文语境：前者用 LLM 给每个 chunk 生成语境前缀（top-20 检索失败率 -49%），后者在 token embedding 层面切块。
2024.10
LightRAG（HKUDS）
砍掉社区检测的轻量图 RAG：双层关键词检索 + 增量更新。→ 第 5 章精读arXiv:2410.05779
2024.11
LazyGraphRAG（微软）
微软对自己重索引路线的纠偏：索引零 LLM 调用（名词短语共现图），LLM 推迟到查询时。索引成本降至 GraphRAG 的 0.1%。
2025.02
Deep Research 引爆
OpenAI Deep Research（HLE 26.6%，约为 o1 的 3 倍）两周后被 Perplexity 打到免费档；同月 NSA/MoBA 稀疏注意力、HippoRAG 2「From RAG to Memory」、A-MEM 记忆论文密集发布。
2025.03
RL 学检索谱系开启
Search-R1 / R1-Searcher / DeepRetrieval：端到端 RL 教模型「何时搜、怎么搜」——检索策略从人写的工作流变成训练出的模型能力。年内经 Kimi-Researcher 到 Tongyi DeepResearch 开源完成闭环。arXiv:2503.09516
2025.04
「RAG is dead」顶点
Llama 4 Scout 标称 10M 上下文；独立测试 128K 文档理解仅 15.6%，争论以「长上下文≠有效上下文」收场。同月 mem0、sleep-time compute 两篇记忆论文发布。
2025.05
Claude Code 弃向量改 grep
「agentic search outperformed everything, by a lot」——代码检索掀起 grep vs embeddings 之争，Cursor/Devin 跟进。
2025.06–09
Context Engineering 学科化
Karpathy 站队（06）→ Manus 实战六原则（07）→ Anthropic 官方指南（09）：「RAG 是 context engineering 的子问题」成为主流叙事；RAGFlow 年终综述称之为「From RAG to Context」。
2025.11
RAG 管道云 API 化
Gemini File Search 与 OpenAI file_search 把「分块-嵌入-检索-引用」整条管道做成托管 API；MCP 捐给 Linux 基金会，检索作为 MCP 工具接入 agent 成默认架构。
2026 H1
吸收与检验
记忆成为平台 API 一等公民（Claude 持久记忆 beta）、多模态检索实用化（Qwen3-VL-Embedding）、1M 上下文 + KV cache 降到 10% 让检索粒度变粗、deep research 评测转向 rubric 化（最强 agent 通过率 <50%）。
展开2025–2026 值得单独一章

这两年的变化密度远超此前五年——逐月编年、RL 学检索谱系、记忆系统演化三条线的完整梳理见第 10 章《2025–2026 编年》（含 2026 上半年实时调研）。

§3史前时代 2017–2021：检索增强是训练问题

这个阶段的所有工作都发生在「LLM 还不够强」的前提下，因此思路是把检索纳入训练，让模型学会利用外部知识：

工作检索方式知识注入方式留给今天的遗产
DrQA 2017TF-IDF / BM25阅读器从检索段落抽取答案两段式「检索→阅读」架构
REALM 2020可学习稠密检索器预训练时端到端联合优化检索器可以被任务信号训练
DPR 2020双塔 BERT + 对比学习—（纯检索工作）embedding 检索范式本身
RAG 2020DPR检索文档作隐变量，与 BART 联合微调「RAG」这个名字
FiD 2021DPR/BM25百段落独立编码、decoder 融合多证据融合的思想
RETRO 2021冻结 BERT 近邻检索chunked cross-attention 层外部知识 ≈ 参数规模的证明

值得注意：DPR 证明「稠密向量打败 BM25」用的是在 QA 数据上训练过的检索器。今天大家拿通用 embedding 模型做检索时 BM25 经常反杀，正是因为丢掉了这个前提——这解释了为什么 hybrid 检索（第 2 章）是生产默认。

§4工程化爆发 2022–2023：Naive RAG 流水线定型

ChatGPT 之后，问题从「怎么训练一个会检索的模型」变成「怎么让一个冻结的、通过 API 访问的 LLM 用上我的私有数据」。答案就是至今仍是所有教程起点的六步流水线：

这个阶段确立的生态分工至今有效：LlamaIndex（原 GPT Index）专注数据索引与检索，LangChain 提供链式编排抽象，向量库负责存储与 ANN 检索。但 Naive RAG 的问题也很快暴露——

检索侧失败

语义相似 ≠ 答案相关；chunk 切断上下文；专有名词/型号匹配不上；多跳问题一次检索拿不全证据。

生成侧失败

检索进噪声导致幻觉；「Lost in the Middle」——上下文中间的证据被注意力忽略；无法回答「全局性」问题（没有任何 chunk 包含答案）。

2023–2024 涌现的所有「高级 RAG」技术——查询变换、混合检索、重排序、GraphRAG、树结构——本质上都是在修补这两类失败。这也是本系列教程的组织逻辑。

§5结构化与前沿 2024–2026：三条修补路线

面对 Naive RAG 的失败模式，行业分化出三条互补的路线：

路线一：把管道每一环做精（Advanced RAG）

hybrid 检索 + RRF 融合、cross-encoder 重排序、contextual retrieval、late chunking、评估驱动迭代。特点是不改变流水线拓扑，只提升每一环的质量。详见第 2 章。

路线二：给语料建结构（Structured RAG）

GraphRAG 建实体图 + 社区摘要（第 4 章）、LightRAG 轻量化图检索（第 5 章）、HippoRAG 用 PageRank 做多跳、RAPTOR/PageIndex 建树（第 6 章）。特点是把检索单位从「文本块」升级为「结构节点」，换取多跳与全局能力，代价是索引成本。

路线三：把决策交给模型（Agentic RAG）

Self-RAG 的按需检索、CRAG 的检索纠错、deep research 的多轮迭代检索、Claude Code 用 grep 式 agentic search 替代向量库。特点是检索次数和策略由模型在推理中决定。详见第 7 章。

视角三条路线不互斥

生产系统往往同时用三条路线：结构化索引（路线二）提供高质量检索工具，工具内部是精调过的管道（路线一），最外层由 agent 决定调用哪个工具、调用几次（路线三）。后面章节的本地项目 gbrain（四层混合检索）和 MiroFish（ReAct 多工具报告）都是这种组合的实例。

§6「RAG 已死」之争：长上下文吃掉检索了吗

每当上下文窗口翻倍，「RAG is dead」就会重现一次。2025 年 Llama 4 Scout 的 10M token 上下文把争论推向顶点。把双方论据摆开：

「已死」论据反驳（实证）
窗口够大，直接把全部文档塞进去「Lost in the Middle」：长上下文中部的证据被注意力稀释，塞得越多找得越差；有效上下文 < 标称上下文
省去索引维护的工程成本典型负载下 RAG 比长上下文便宜 8–82 倍且延迟更低（Contextual AI 实测）——每次请求重复计费全量 token 无法摊销
模型自己会在上下文里「检索」企业语料远超 10M token；权限控制、数据新鲜度、审计溯源都需要检索层

更准确的结论是：死掉的是「向量库 = RAG」的固定等式，活下来的是检索本身。检索正在降级为 agent 上下文工程（context engineering）中的一个工具——有时是向量检索，有时是 grep、SQL、图遍历或树搜索。RAGFlow 2025 年终综述称之为「From RAG to Context」。这个判断是理解后续所有章节的透镜。
