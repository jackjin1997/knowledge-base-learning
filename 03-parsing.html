Chapter 03 · Document Parsing & OCR

文档解析与 OCR：RAG 的第一公里

改进解析带来的收益，常常大于换一个更强的 LLM。本章讲清 PDF 为什么难、管线派与 VLM 派的机制差异、以及 2024–2026 最重要的格局变化：1B 级专用视觉模型全面超越管线方案。

管线派 MinerU / Docling VLM 派 olmOCR / dots.ocr 标尺 OmniDocBench 对照 ColPali 免解析

§1为什么文档解析是 RAG 质量瓶颈

PDF 的本质是「打印指令」格式，不是「结构化内容」格式。它内部只有「在坐标 (x, y) 处以某字体绘制某字符」的绘图指令——没有段落、没有标题、没有表格、没有阅读顺序。解析器必须依靠几何启发式或模型反向推断逻辑结构。四个具体难点：

布局是二维的，LLM 输入是一维的

双栏论文按字符流顺序读出来会左右两栏交错。阅读顺序（reading order）本身就是一个需要模型解决的任务。

表格退化

无边框表、合并单元格、跨页表在文本流中变成无法对齐的碎片，行列关系彻底丢失。

公式乱码

数学符号在 PDF 里是特殊字体的字形，直接抽取得到乱码，必须识别转成 LaTeX。

扫描件无文本层

整页只是一张图片，任何文本抽取都必须先 OCR。

§2管线式方案：每个子任务一个专用模型

管线派的思路：布局检测、文本抽取、表格识别、公式识别各用专门模型，最后按坐标重组。可解释、可局部替换，但误差级联——布局检测框错一个区域，后面全错。

工具机制强项短板
PyMuPDF直接读 PDF 文本层，按坐标提取毫秒级/页，零模型无布局理解；扫描件无能为力
pdfplumberpdfminer 基础上用线条/坐标几何规则抽表有边框规整表格（财报）无边框表基本失效
unstructured多格式统一入口；hi_res 模式 = detectron2 布局 + Tesseract/PaddleOCR生态整合（LangChain/LlamaIndex 原生）泛而不精，hi_res 慢
Docling（IBM）DocLayNet 布局分析 + TableFormer 表格结构识别（视觉 Transformer 预测行列/表头/合并单元格）表格结构最强之一；MIT 协议纯本地；统一 DoclingDocument → MD/JSON公式偏弱
marker布局检测 + texify 系公式模型，输出 Markdown速度/质量平衡，学术友好模糊扫描/水印页出错；表格 TEDS 偏低
MinerU（上海AI Lab）DocLayout-YOLO 布局 + YOLOv8 公式检测 + UniMERNet 公式识别 + PaddleOCR + 阅读顺序后处理中文突出；长期管线派最强（布局检测是关键优势）组件多部署重；误差级联

本地关联：LightRAG 的 parser 层直接集成了 MinerU 和 Docling 作为可选解析后端（第 5 章）。

§3端到端 VLM 浪潮：图像进，Markdown 出

2023 年起的范式转变：不再「检测→识别→重组」，而是一个视觉语言模型端到端完成——整页图像输入，结构化 Markdown/HTML 输出。按时间线看关键节点：

2023
Nougat（Meta）
先驱：Swin 编码器 + mBART 解码器，学术 PDF 直出含 LaTeX 的 Markdown。局限：只擅长论文版式，易重复退化。arXiv:2308.13418
2024
GOT-OCR2.0
「OCR-2.0」理念：文字、公式、表格、图表、乐谱统一视为「字符」。仅 580M 参数。弱点：彩色背景页失败。arXiv:2409.01704
2025
olmOCR（AllenAI）
Qwen2-VL-7B 微调，核心技巧 document anchoring——把 PyPDF 抽出的文本块坐标注入提示，给视觉识别提供文本层锚点。成本约 $176–190/百万页（GPT-4o 的 1/32）。配套 olmOCR-Bench 用 7000+ 单元测试式判分取代编辑距离。arXiv:2502.18443
2025
dots.ocr（小红书）
1.7B 单一 VLM 同时做布局检测与内容识别，换 prompt 切换任务，消除多模型协调。OmniDocBench 文本/表格/阅读顺序三项 SOTA。arXiv:2512.02498
2025
PaddleOCR-VL（百度）
仅 0.9B（NaViT 动态分辨率 + ERNIE-4.5-0.3B），OmniDocBench v1.5 以 92.6 登顶，109 种语言——证明「小模型 + 好数据」路线可行。
2025.09
MinerU2.5
管线派转向 VLM 的代表：1.2B「解耦两阶段」——先在 1036² 下采样图上做全局布局，再按布局裁出原生分辨率 patch 做细粒度识别，避免整页高分辨率的 token 爆炸。arXiv:2509.22186
2025.10
DeepSeek-OCR：醉翁之意不在 OCR
立意是「上下文光学压缩」：用 2D 视觉 token 压缩长文本。压缩比 <10× 时 OCR 精度 97%，20× 仍有 60%；100 个视觉 token 就超过用 256 token 的 GOT-OCR2.0；单卡 A100 日处理 20 万+页。它暗示：文档图像可能成为 LLM 长上下文/记忆的压缩载体。arXiv:2510.18234
避坑官方 benchmark ≠ 真实文档

Mistral OCR（2025 商用 API）官方宣称 94.9% 准确率、$1/千页；但独立评测显示发票等真实场景可靠性仅 63.75%，纯转写准确 98.75% 的同时漏抽 27.5% 的数据。任何 OCR 选型都要在自己的文档样本上跑过再定。

§4OmniDocBench：统一标尺与格局

OmniDocBench（上海AI Lab，CVPR 2025）：1651 页、10 类文档（论文/教材/试卷/财报/报纸/杂志/幻灯片等）、5 种布局、5 种语言。综合分 = ((1−文本编辑距离)×100 + 表格 TEDS + 公式 CDM) / 3。v1.6 榜单（2026 年中）的格局非常清晰：

格局2024–2026 最重要的变化

1B 级专用 VLM 已全面超过管线方案和百倍大的通用 VLM。专用模型比 GPT-4o 高近 10 分、比自家管线版高 9 分。含义：新项目做文档解析，默认起点应该是小型专用 VLM 而不是拼管线，也不是调通用大模型 API。（榜单数字随版本变动，引用时注明基准版本。）

§5难点专题：表格、公式、图表、阅读顺序

难点技术路线评测指标工程要点
表格结构识别模型（TableFormer/SLANet 输出 HTML/OTSL）vs VLM 直出 HTMLTEDS（树编辑距离）喂 LLM 时 HTML/Markdown 显著优于纯文本（保留单元格对齐）；跨页表合并至今是各家痛点，chunk 时表格要整块保留
公式专用模型（UniMERNet/pix2tex）或 VLM 直出 LaTeXCDM（字符检测匹配）学术 RAG 中公式丢失会让 chunk 语义残缺
图表chart-to-table / chart-to-JSON（GOT-OCR2.0、PaddleOCR-VL 支持）—不反解的话，图表信息在纯文本 RAG 中完全丢失
阅读顺序规则（XY-Cut / XY-Cut++）vs VLM 训练中隐式学习独立评分维度双栏、图文混排、报纸版式的核心难题

§6商业 API 速览

Azure Document Intelligence / AWS Textract

老牌企业级：预置模型（发票/收据/身份证）+ 自定义训练，合规完善。适合表单类结构化抽取，深度绑定各自云生态。

LlamaParse

面向 RAG 设计，多档解析模式（含 agentic 模式调 VLM），LlamaIndex 技术栈里从 PDF 到 Markdown 路径最短。

Reducto

「可审计的高精度」：agentic OCR 纠错层 + 字段级引用回溯到原文位置，SOC 2 / HIPAA / 零数据保留。金融法务首选。

Chunkr

开源可自托管的「解析 + 分块」一体化服务，直接输出 RAG-ready chunks。

选择逻辑：合规可审计 → Reducto / Azure DI；LlamaIndex 栈 → LlamaParse；成本敏感大批量 → 自托管开源 VLM。

§7选型决策树

§8与 RAG 的衔接 + ColPali 对照路线

解析输出格式决定 chunking 上限

  • 输出带层级的 Markdown/JSON，才能做 markdown-aware chunking（按标题/结构边界切）——实证比固定长度和语义切分高 5–10 个百分点（Snowflake 金融 RAG）。
  • parser 和 chunker 要协同选择：结构感知的 chunker 只有配上保留布局的 parser 才有价值。
  • 保留元素类型和页码/坐标元数据：表格整块保留、引用溯源、chunk 过滤都靠它。

对照路线：ColPali——不解析，直接视觉检索

ColPali（arXiv:2407.01449）把整页文档图像经 VLM 编码为 patch 网格，每个 patch 投影成 128 维向量，用 ColBERT 式 late interaction 与查询匹配——完全绕过 OCR、解析、chunking 三步。在 ViDoRe 基准上 NDCG@5 达 0.81，传统「解析→切分→嵌入」管线只有 0.66。

解析路线ColPali 视觉路线
视觉密集文档图表信息丢失碾压式领先
存储单向量/块多向量/页，开销大
检索产物文本，可全文操作、细粒度引用页面图片，需多模态 LLM 消费
混合架构实践折中：ColPali 做页面级召回 + VLM 对召回页面即时解析
承上启下本章之后

解析把文档变成干净的结构化文本，第 2 章的管道把它变成可检索的索引。接下来三章讨论的问题是：当「文本块 + 向量」这个索引单位不够用时，如何给语料建图（第 4、5 章）或建树（第 6 章）。
