PageIndex 精读：不用向量，让 LLM 像人一样翻目录

领域专家找答案不算余弦相似度——先看目录，推理该翻哪章，再翻到那几页。PageIndex（Vectify AI）把这个过程变成系统：文档 → 目录树，检索 → LLM 树搜索。零向量库。

范式 reasoning-based retrieval 本地 ../PageIndex 声称 FinanceBench 98.7% 形态 树 + 三个工具

§1核心论点：相似度 ≠ 相关度

向量检索假设「语义相似的文本是相关的」。这个假设在两类场景系统性失效：

  • 专业长文档：一份 300 页年报里，「流动性风险」在 20 个章节都出现，语义都相似——但审计师知道答案只在「财务报表附注 7」。判断依据是文档结构与领域知识，不是文本相似度。
  • 需要推理的查询：「这家公司的表外负债有多少」需要知道该看哪些附注、怎么交叉对照——这是推理链，不是最近邻查找。

PageIndex 的方案：索引期把 PDF 解析成保留原生层级的目录树（节点 = 章节，带页码范围和摘要）；查询期让 LLM 读树结构、推理该进入哪个分支、取回那几页原文再回答。整个过程没有 embedding，没有向量库，没有 chunking。

定位它是第 1 章五族中的「树族 + Agentic 族」交点

树提供结构（文档自身的包含关系），agent 提供检索策略（LLM 决定往哪走）。对照第 4、5 章：图 RAG 把「跨文档关联」结构化，PageIndex 把「单文档层级」结构化——它目前只支持单文档检索，多文档需要外层再套一层文档选择逻辑（仓库 examples/tutorials/doc-search/ 给了 metadata / semantics / description 三种策略）。

§3递归细分与节点摘要

大节点细分（process_large_node_recursively() page_index.py:1000）：一个「第 3 章」可能占 80 页，直接作为叶子太粗。当节点同时满足「页数 > 10 且 token ≥ 20000」（config.yaml 的 max_page_num_each_node / max_token_num_each_node）时，对该节点内容重新跑一遍 meta_processor() 生成子树，递归进行。双条件设计避免了「页数多但内容稀」的表格章节被无谓切碎。

节点摘要（可选，if_add_node_summary: yes）：树建好后扁平化所有节点，asyncio.gather 并行为每个节点生成一句「本节讲什么」的摘要——这是检索期 LLM 判断「该进哪个分支」的主要依据：

pageindex/utils.py:578-596async def generate_summaries_for_structure(structure, model=None):
    nodes = structure_to_list(structure)                # 树 → 扁平列表
    tasks = [generate_node_summary(n, model) for n in nodes]
    summaries = await asyncio.gather(*tasks)            # 全节点并行
    for node, summary in zip(nodes, summaries):
        node['summary'] = summary                       # 写回树
    return structure

最终产物是一个 JSON 树（results/{doc}_structure.json）：每个节点有 title、start/end_index（页码范围）、summary、子节点。默认不存原文（if_add_node_text: no）——原文检索期按页码现取，树本身保持轻量。

§4检索：Agent 拿三个工具在树上推理

检索侧没有「检索算法」，只有一个带三个工具的 agent（examples/agentic_vectorless_rag_demo.py）。这就是第 1 章说的「检索 = agent 的工具调用」的最纯粹形态：

get_document_structure()

retrieve.py:100 — 完整树结构，剥掉 text 字段省 token。LLM 读它决定去哪。

retrieve.py:110 — 按页取原文，支持 "5-7"、"3,8"、"12" 三种格式。

examples/agentic_vectorless_rag_demo.py:44-52（agent 系统提示）You are PageIndex, a document QA assistant.

TOOL USE:
- Call get_document() first to confirm status and page/line count.
- Call get_document_structure() to identify relevant page ranges.
- Call get_page_content(pages="5-7") with tight ranges;
  never fetch the whole document.

Answer based only on tool output. Be concise.

四步循环：确认规模 → 读树推理相关章节 → 紧密页码范围取原文（提示词明令禁止整本拉取）→ 基于原文作答。每次调用前强制输出一句理由——检索路径完全可审计：答案错了能看到它在哪一步走偏。

细节为什么树结构要剥掉原文

树（标题 + 摘要 + 页码）是 LLM 的「地图」，通常几千 token 就能装下几百页文档的骨架——这让单次上下文能容纳完整全局视野，这正是向量 RAG 永远给不了的东西：chunk 看不到自己在文档中的位置，树节点天生知道。

§5对照：RAPTOR 与树族谱系

树族有两条支线，结构来源完全不同：

RAPTOR（ICLR 2024）PageIndex
树从哪来自建：embedding → GMM 软聚类 → LLM 摘要每簇 → 递归向上文档原生：TOC / 章节层级直接映射
检索方式collapsed tree 上做向量检索（仍是相似度）LLM 推理下钻（无向量）
成本重心索引期（每簇一次 LLM 摘要）查询期（每问多轮 LLM 推理）
擅长「综合理解型」问题命中摘要层；QuALITY 绝对提升 20%结构清晰的专业文档精确定位；FinanceBench 98.7%（官方声称）
依赖聚类质量文档要有可提取的结构；无结构文档退化到路径 C 自建树

与图族的本质区别（承接第 4、5 章）：树是「包含关系」（单文档内的抽象层级），图是「关联关系」（跨文档的任意拓扑）。多跳跨文档问题树帮不上忙；单文档内的层级定位图也不擅长。

§6与向量 RAG 的取舍

维度PageIndex向量 RAG
检索机制LLM 推理 + 树遍历ANN 相似度
基础设施零向量库，一个 JSON 文件embedding 服务 + 向量库
延迟高——每问 3–5 次 LLM 调用毫秒级 ANN + 1 次 LLM
单次查询成本高（多轮推理 token）低
可解释性完整推理路径 + 页码引用相似度分数黑箱
多轮对话对话历史自然作为检索上下文需要重写 query
跨文档需外层文档选择逻辑统一向量空间原生支持
判断什么时候选它

文档长、专业、结构良好（财报、法规、技术手册），查询高价值、容忍秒级延迟（审计、尽调、合规问答）——这是 PageIndex 的主场，官方案例 Mafin 2.5 做的正是金融文档。海量短文档、高 QPS、低延迟场景请回第 2 章的管道。它的成本结构与 GraphRAG 正好互为镜像：GraphRAG 索引贵查询廉，PageIndex 索引廉查询贵——「你的查询量摊得平哪边」是最实际的选型标准。
