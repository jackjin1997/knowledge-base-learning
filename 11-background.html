自维护知识库：Wiki、做梦与睡眠计算

前十章描述的一切——解析、检索、图树索引、agentic 调度——都发生在「一次请求的生命周期内」。本章讲请求之间的事：没人提问的时候，知识自己在被综合、体检、重组和遗忘。这一层在 2025–2026 从散点收敛成了体系，Anthropic 给它起的名字叫「做梦」。

思想 Gwern Daydreaming 机制 Letta Sleep-time 产品 Claude Dreaming 底座 Agentic Wiki

§1论点：知识系统有两个时钟

把本教程讲过的所有技术按「什么时候运行」分类，会发现它们全挤在一个时钟上：

请求时（query-time）请求之间（idle-time）
发生什么解析、检索、融合、重排、生成——第 2–6 章的全部综合、去重、解决矛盾、蒸馏技能、遗忘过期事实
索引期算什么GraphRAG 社区报告、RAPTOR 摘要树、contextual retrieval 是「写入时综合」——但都是一次性的，建完不再自我更新这一层是持续的：知识库像生物一样有新陈代谢
成本模型延迟敏感，用户在等闲时算力，便宜、可批量、可中断
2026 的代表本教程第 2–6 章agentic wiki 的 lint、Letta sleep-time、Claude Dreaming
核心为什么这是独立的一层

人脑的类比很直接：白天经历（请求时），夜间海马体回放并把短期记忆巩固到新皮层（闲时）。第 7 章的 HippoRAG 借了海马体的检索机制，这一章的系统借的是海马体的巩固机制。没有夜班的知识库只会越积越乱——chunk 冗余、事实互相矛盾、昨天综合出的认识今天重算一遍。第 10 章 Karpathy 对 RAG「没有积累」的批评，根子就在缺这一层。

§2Wiki 一族：自维护的知识底座

第 10 章 §6 讲了 wiki 热潮的来龙去脉（DeepWiki 产品线 + Karpathy llm-wiki），这里补上体系视角：wiki 是「后台认知」最朴素的载体——三个操作里 ingest 和 lint 本来就是后台作业。查证结论：「agentic wiki」不是某个明星项目，而是 llm-wiki 之后涌现的一个品类，值得看的两个实体：

Onyx agent-wiki（YC）

最产品级的实现：Markdown + 文件系统 + Git 三件套，agent 经 MCP 读写；页面可携带 update policy（「保持简洁」「不许动 SLA 表」）——把「哪些知识允许 agent 自动改写」变成页面级配置，直面自噬风险。

agentwiki.org

自述受 llm-wiki 启发：raw sources 被拆成原子页面 + 语义嵌入交叉引用（6400+ 页），提供 JSON-RPC 读写接口——一个「为 agent 而非人类读者设计」的公共 wiki 样本。

与本地仓库的呼应：gbrain 的零 LLM 链接抽取（第 8 章 §1）解决的是「wiki 的边怎么来」；RAGFlow 的 memory 模块（第 8 章 §3）如果长出页面和双链，就是一体化引擎补上这一层的信号。

辨析与 PageIndex 的区别：都去向量化，但一个是索引、一个是笔记

两者共享「去向量化」表皮——都靠结构 + LLM 推理做导航式检索而非相似度匹配。但内核相反：PageIndex 是给别人写的文档建目录卡——树是从文档无损派生的只读索引，知识永远在原文里，回答时取回原文逐字引用（可溯源到页码，所以敢打财务/法律）；wiki 是自己记笔记——页面本身就是知识，被 LLM 持续改写再创作，原文只是归档原料。三个推论：① 可信度模型相反——要权威引用选 PageIndex 路线，要认知复利选 wiki 路线（代价是自噬风险，见 §7）；② 时钟不同——PageIndex 全在请求时，wiki 的本体（ingest/lint）在请求之间；③ 去向量化只是副产品，agentwiki.org 给页面加语义嵌入并不改变其笔记本性。两者可组合：wiki 的 raw/ 层用 PageIndex 树管理——笔记引用目录卡，目录卡指向原文，这个组合尚无产品化实现。

§3Daydreaming：思想源头（Gwern, 2025.07）

「做梦」这条线的概念源头是 Gwern 的长文《LLM Daydreaming》。他的出发点是一个尖锐观察：LLM 知识渊博却从未产出真正的科学突破，因为缺三样人类有的东西——从经验学习的能力（权重冻结）、持续的后台思考、以及「默认模式网络」（DMN，人走神时自发产生洞见的脑区）。

两个反直觉的论断使这篇文章超出「又一个 agent 架构」：

  • Daydreaming tax：绝大多数概念组合一无所获（他估计有效洞见比例 20:1 甚至更差），开启 DDL 可能使成本高 200 倍。这反过来解释了当前 LLM 为何「经济」——它们靠回避新颖性生成实现商业可行。
  • 数据墙的出路：昂贵的「做梦 AI」为下一代模型生成专有训练数据，且构成对抗蒸馏的护城河——有价值的洞见来自「没人会想到去问」的组合，蒸馏者无从模仿。

§4睡眠计算：机制验证（Letta, 2025.04 → 2026.06）

Gwern 提供思想，Letta（MemGPT 团队）提供了可测量的机制。《Sleep-time Compute》论文（arXiv:2504.13171）的核心结果：离线预思考上下文，达到同等准确率所需的 test-time 算力降约 5 倍——把「夜班有用」从直觉变成了数字。

产品架构是一个干净的职责分离：

# Letta sleep-time 双 agent 架构（docs.letta.com）
primary agent     面向用户实时对话 —— 但无权编辑自己的核心记忆
sleep-time agent  后台异步运行，独占记忆编辑工具：
                  整合碎片记忆 · 跨会话找模式 · 去重 · 归档过时信息

「对话的 agent 不许写记忆」这个约束值得细品：它把记忆质量从「对话副产品」变成「专门进程的职责」——正是操作系统里前台进程与 GC 的分工。2026.06 Letta 更进一步发布《Memory Models: Towards Agents That Learn》：训练专门在睡眠期生成记忆的模型（memory-native RL），并正式把这个过程命名为 agent dreaming，同时给出一个激进预言：「最终世界上绝大多数推理负载将由 sleep-time compute 驱动。」

§5Dreaming 产品化（2026 上半年）

你印象里「做梦很火」的那波热度，主要来自 2026 年 5 月 Anthropic 的动作——这一层从论文变成了旗舰产品功能：

系统机制备注
Claude Dreaming / Dreams
（Anthropic，2026.05.06 发布）		定时离线读取记忆库 + 最多 100 个历史会话 transcript → 重组记忆：去重合并、矛盾按最新值解决、剪除失效引用。官方明确类比 REM 睡眠记忆巩固research preview；法律 AI 公司 Harvey 称开启后任务完成率提升约 6 倍（厂商自报，未独立验证）
Claude Code「AutoDream」第三方描述：四阶段（orientation → gather signal → consolidation → prune & index），维持 MEMORY.md ≤200 行，每 24 小时且 5+ 会话后触发来源为第三方博客与社区复刻 skill，非官方文档，细节谨慎采信
Auto-Dreamer（学术，arXiv:2605.20616）基于互补学习系统（CLS）理论：会话内快速记忆获取与跨会话慢速巩固解耦，学一个离线巩固器执行多步工具调用（搜记忆→溯源轨迹→合成新条目）ScienceWorld 超最强基线 7 分且活跃记忆库小 12 倍；零迁移到 ALFWorld/WebArena
生态扩散Xiaomi MiMo Code 内置每周 dream cycle；NousResearch hermes-agent 有 Auto Dream 需求单从旗舰到开源工具链的常规扩散路径
辨析此 Dreamer 非彼 Dreamer

搜「dream + AI」会撞上 Hafner 的 Dreamer 世界模型谱系（DreamerV3 → Google Genie 3 / Project Genie，Waymo 用它模拟极端路况）——那是 RL 里「在想象的轨迹中训练策略」，与 agent 记忆巩固是不同谱系。两条线目前只在「在想象中学习」的概念层面有交汇（如 Qwen-AgentWorld 的语言世界模型——这个联系属推断）。

§6学术谱系与命名现状

这一层的学术工作在 2025–2026 密集成形，按目的分三支：

分支代表工作做什么
睡眠巩固Auto-Dreamer（2605.20616）、Do LMs Need Sleep?（2605.26099）、TiMem 时序分层巩固（2601.02845）把碎片经验重组为持久记忆
经验 → 技能Voyager（技能库开山）、ExpeL（轨迹→自然语言 insight）、Memp 程序性记忆蒸馏（2508.06433）、MemSkill（2602.02474）、ExpWeaver（2606.01041）把「做过的事」蒸馏成「会做的事」——闲时把轨迹变成可复用技能
遗忘Learning to Forget（2603.14517）；工程侧：RAGFlow memory 的 forget_at 字段、gbrain 软删除知识库的减法——不遗忘的记忆最终不可检索

建制化证据：ICLR 2026 已出现 MemAgents workshop 提案。命名现状：尚无统一学科名——最接近通名的是「sleep-time compute」（论文原词）和「agent dreaming」（Letta + Anthropic 双重背书，2026 上半年的事实通称）；「background cognition」未固化。本章标题用「自维护知识库」取其功能而非流行语。

§7体系归位：四层楼 + 夜班

把这一层放回学习地图（index 的全景图已同步更新）：

三点推论：

  • GraphRAG 和 RAPTOR 是「只跑一次的夜班」——社区报告和摘要树本质是离线综合，只是没有持续更新机制。LazyGraphRAG 把综合推回请求时，dreaming 系把综合变成常驻进程：三者是同一权衡轴上的三个点。
  • 夜班的风险是自噬（第 10 章 §6 的 Nature 2024 论据）：LLM 反复改写自产内容会漂移。工程护栏已经出现——wiki 的 lint、Onyx 的页面级 update policy、Anthropic 的「矛盾按最新值解决」都是对策。
  • 成本叙事翻转：第 4 章批评 GraphRAG 索引贵，但那是按「一次性建库」算账；如果 Letta 的预言成立（多数推理负载转向 sleep-time），闲时批量算力的定价逻辑将重塑「索引期 vs 查询期」这根贯穿全书的权衡轴。

§8体系边界声明：本教程仍未覆盖的

重审全书后，如实列出体系之外的三块（宁可承认没讲，不装作全覆盖）：

缺口是什么为什么重要
结构化数据检索NL2SQL / 表格 QA / semantic layer——从数据库而非文档中检索企业知识库常常是「文档 + 数据库」双源；RAGFlow、Coze 都内置 NL2SQL。与本书文档检索是并列的另一根支柱
RAG 安全检索内容携带的间接 prompt 注入、权限感知检索（谁能检回什么）、数据渗出生产部署的必答题——检索给了外部内容进入 prompt 的通道，也给了攻击者
Serving 基础设施KV-cache 管理（Mooncake，第 9 章一笔带过）、embedding 服务化、索引热更新的工程细节决定同一套架构的成本差一个数量级

需要展开哪一块，可以按第 9、10 章的方式做专项调研补章。
